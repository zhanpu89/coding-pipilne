# 特征交互图

五个核心特征的闭环协作关系：

```
     ┌──────────────────────────────────────────────────┐
     │                  OODA 元认知                       │
     │  每个 Phase 前后: Observe→Orient→Decide→Act      │
     └────────┬────────────────────────────┬────────────┘
              │  Orient 阶段评估           │ 后置 OODA 发现问题
              ▼                            ▼
     ┌────────────────┐          ┌──────────────────┐
     │ 置信度驱动策略   │          │  跨 Phase 反馈    │
     │ 🟢高→标准流程   │          │ 🔴阻断→回溯修复  │
     │ 🟡中→增自检    │          │ 🟡影响→最终统一  │
     │ 🔴低→Spike探针 │          │ 🟢参考→Step 7展示 │
     └────────┬────────┘          └────────┬─────────┘
              │                            │
              ▼                            ▼
     ┌───────────────────────────────────────────────┐
     │          自适应恢复矩阵                         │
     │  🅰崩溃→精简重试  🅱评审→定向修复               │
     │  🅲质量→重审需求  🅳死循环→搜索记忆换策略       │
     └──────────────────────┬────────────────────────┘
                            │ Step 7 触发
                            ▼
     ┌───────────────────────────────────────────────┐
     │          运行后自进化审查                       │
     │  扫描决策 → 识别模式 → 产出可复用建议           │
     │  → 改进 SKILL.md / rules（经用户确认后落地）    │
     └───────────────────────────────────────────────┘
```

**典型交互场景：**
1. OODA Orient 判断置信度低 → 触发 Spike → Spike 成功后置信度提升 → 走标准流程
2. 后置 OODA 发现产出物有偏差 → 触发跨 Phase 反馈 🔴 阻断型 → 回溯修复 → 继续
3. 评审失败 → 🅱 恢复 → 修复时发现根因在更早 Phase → 转跨 Phase 反馈 🔴
4. Step 7 自进化审查 → 发现"评审频繁发现验证缺失"模式 → 建议改进 code-discipline.md
