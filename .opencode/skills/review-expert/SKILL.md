## 核心原则

本 agent **只读文件内容**，评审依据仅为文档交付物的客观内容与参考契约文件。不知道也不推测作者是谁、有何创作意图、做了哪些设计推理。

## 风险等级

| 等级 | 阻断 |
|------|------|
| P0：崩溃/数据丢失/资金/安全 | 是 |
| P1：功能缺陷/单点/性能 | 生产版阻断，MVP 降 P2 |
| P2：可维护性/规范 | 否 |

## 结论

✅ 无P0且P1≤2（全流程≤6）| ⚠️ 无P0但P1超阈值 | ❌ 有P0

## 工作流

**Step 0：** 确定项目阶段（MVP/生产版）→ 确定评审模式（需求/架构/详细设计/多端/测试用例/全流程）→ 加载 `resources/check-common.md` → 就绪检查不通过则返回。

**Step 1：** 标记 3~5 个高风险点（接口缝隙/异常处理/并发事务/外部依赖/不可逆操作）。文档 <500 字或全占位符则停止。

**Step 2：** 按模式加载检查清单：
- 需求 → `resources/check-req.md`
- 架构 → `resources/check-arch.md`
- 详细设计/多端 → `resources/check-detailed.md`（先比对 `doc/arch/` 和 `doc/prd/`，缺失=P0；多端做端间接口对齐）
- 测试用例 → `resources/check-test.md`

信息缺失 >30% → ❌ 不通过。

**Step 3（仅全流程）：** 因果链追溯（实现→需求 和 需求→实现），发现断层则列出。

**Step 4：** 加载 `templates/report-template.md` 按模式跳转对应节。输出 `doc/review/{项目/模块名}_{类型}评审报告.md`。

### 熔断

被评审文档不存在 / 就绪检查不通过 / 模式无法确定 / 全占位符 → 停止。

### JSON 写入安全

出现 `JSON parsing failed` 时，说明工具调用 payload 格式有误。写入大文件时分多次 `write` 调用，每次不超过 2000 字符。
