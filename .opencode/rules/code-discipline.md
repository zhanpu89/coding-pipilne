# 编码纪律规则 (CODE DISCIPLINE RULES)

**适用范围：所有代码生成和修改场景。** 来自 CLAUDE.md 的行为准则，减少常见 AI 编码错误。

## 原则 1：先思考再编码

**不要假设。不要隐藏疑惑。摊开权衡。**

实现之前：
- 明确陈述你的假设。不确定就问。
- 如果有多种解释，列出它们——不要默默选一个。
- 如果有更简单的方法，说出来。必要时可以反驳。
- 如果某件事不清晰，停下来。指出什么让人困惑。提问。

## 原则 2：简洁优先

**用最少的代码解决问题。不做投机性设计。**

- 不多实现任何未被要求的功能
- 不为单次使用场景做抽象
- 不添加未被要求的"灵活性"或"可配置性"
- 不为不可能发生的场景写错误处理
- 如果你写了 200 行但可以写成 50 行，重写

问自己："资深工程师会说这个过度设计了吗？"如果是，简化。

## 原则 3：手术式修改

**只触碰你必须碰的。只清理你自己造成的混乱。**

编辑既有代码时：
- 不要"改进"无关的代码、注释或格式
- 不要重构没坏的东西
- 匹配既有风格，即使你会用不同的做法
- 如果你注意到无关的僵尸代码，提一句——但不要删

你的修改产生了孤立的引用时：
- 移除**你的修改**导致的未使用的导入/变量/函数
- 不要删除既有僵尸代码，除非被要求

检验标准：每一行改动的代码都应当直接追溯到用户的需求。

## 原则 4：目标驱动执行

**定义成功标准。循环直到验证通过。**

把任务转化为可验证的目标：
- "添加校验" → "为非法输入写测试，然后让它们通过"
- "修复 bug" → "写一个能复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后的测试都通过"

多步骤任务时，列出简要计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
```

强成功标准让你可以独立循环。弱标准（"让它工作"）需要不断澄清。
