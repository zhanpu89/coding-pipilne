# 文档对齐规则 (DOC ALIGNMENT RULES)

**适用范围：增量需求 / Bug 修复场景。** 代码与文档同步修改，杜绝设计与代码漂移。

## 原则：同步修改

修改代码的同时必须更新对应文档，二者在同一任务中完成。不允许只改代码不改文档。

## 识别文档位置

精确定位到模块后，同步识别对应文档位置：

| 代码变更 | 对应文档 |
|---------|---------|
| 新增/修改 API 端点 | `doc/detailed/{模块}_*.md` — OpenAPI 3.0 章节 |
| 新增/修改业务逻辑 | `doc/detailed/{模块}_*.md` — 业务规则章节 |
| 修改字段/模型/DDL | `doc/detailed/{模块}_*.md` — DDL 章节 + `doc/db/` |
| 修改异常/错误码 | `doc/detailed/{模块}_*.md` — OpenAPI 错误码 |
| 新增/修改接口契约 | `doc/detailed/` 跨模块接口清单 |
| 修改项目规则/编码规范 | `doc/detailed/项目规则.md` / `编码规范.md` |
| 新增/修改测试场景 | `doc/tester/{模块}_测试用例.md` |

## 更新要求

1. **文档版本递增** — 每次文档修改后版本号递增（如 v1.0.0 → v1.0.1）
2. **变更记录追加** — 在文档末尾变更表中追加条目：`| v1.0.1 | {日期} | {变更描述} | 作者 |`
3. **代码同步提交** — 代码 PR 中必须包含对应文档的变更，作为同一个 commit 或同组 commits

## 验证标准

- [ ] 对应文档已更新，与最终代码一致
- [ ] 文档版本号已递增（patch 级别）
- [ ] 变更记录已追加
- [ ] 新增接口/字段在 OpenAPI 中有完整定义
- [ ] 新增业务规则在详设中有完整描述
